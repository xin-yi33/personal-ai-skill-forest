## 摘要

随着大语言模型（LLM）从单轮问答向长程任务执行转型，工具与技能调用已成为能力扩展的主要载体。然而，当 AI 助手管理数百乃至数千个工具时，"工具膨胀"问题演变为结构性障碍：Anthropic 的工程报告显示，仅 5 个 MCP 服务器（约 58 个工具）即消耗约 55K tokens，极端场景可达 134K tokens。本文提出"个人 AI 技能森林"架构，基于多棵并行 B+ 树，系统性解决技能管理在规模化场景下的五大核心挑战：可扩展性、个性化、可进化性、可解释性与 Token 效率。该系统由 12 个互锁机制（M1–M12）组成，分为三层：森林级路由层（M2）将查询导向领域专属 B+ 树；核心操作机制组（M4 依赖回溯、M5 多候选选择、M6 参数合并、M7 私人技能遮蔽、M9 角色降维）对复杂推理任务进行结构化预处理；双层自进化系统（行动反思与思维反思）实现技能的持续积累。我们在包含 5 个领域、5,000 个 API 的合成 ToolBench 风格数据集上开展了六组综合实验。实验结果表明：（1）森林架构在 Accuracy@5 上提升 8.8%，MRR 提升 28.8%；（2）端到端 Token 消耗降低 79.3%（612→127 tokens）；（3）消融实验确认每个机制的独立贡献，其中依赖回溯（+0.592 链完整率）和参数合并（+0.533 冲突消解率）影响最大；（4）双层反思机制在三轮学习后将任务成功率提升 21.6 个百分点，步骤数减少 41.6%；（5）Token 节省比例在 500 至 5,000 个 API 的规模范围内稳定在约 82%。上述结果确立了技能森林作为可扩展、个性化、自进化 AI 技能管理范式的可行性。

**关键词**：技能管理；B+ 树索引；智能体；工具检索；自进化；大语言模型；个性化

## Abstract

As large language models (LLMs) transition from single-turn question answering to long-horizon task execution, tool and skill invocation has become the primary vehicle for capability extension. However, when an AI assistant manages hundreds or thousands of tools, the "tool inflation" problem emerges as a structural barrier. This paper proposes the Personal AI Skill Forest, a novel architecture based on multiple parallel B+ trees that addresses five core challenges in skill management at scale: scalability, personalization, evolvability, explainability, and token efficiency. The system comprises 12 interlocking mechanisms organized into three layers: a forest-level routing layer, core operational mechanisms (dependency tracing, multi-candidate selection, parameter merging, private skill masking, role reduction), and a dual-layer self-evolution system. Experiments on a 5,000-API dataset demonstrate: (1) 8.8% higher Accuracy@5 and 28.8% higher MRR; (2) 79.3% token reduction; (3) each mechanism contributes independently; (4) 21.6pp success rate improvement through self-evolution; (5) stable ~82% token savings across scales.

**Keywords**: skill management; B+ tree indexing; intelligent agents; tool retrieval; self-evolution; large language models; personalization

## 1. 引言

大语言模型（LLM）的快速发展催生了从单轮问答向长程多步任务执行的范式转变（Wei et al., 2022; Yao et al., 2023）。在这一新兴范式中，工具与技能调用已成为 AI 智能体超越文本生成、扩展能力边界的主要机制（Schick et al., 2023; Qin et al., 2023）。模型上下文协议（MCP）及 LangChain（Chase, 2022）、AutoGen（Wu et al., 2023）、LangGraph 等智能体框架已将工具集成形式化，使智能体能够与外部 API、数据库和服务交互。

然而，随着可用工具生态的扩展，一个关键瓶颈浮现：**工具膨胀**。Anthropic 关于高级工具使用的工程报告显示，仅 5 个 MCP 服务器（约 58 个工具）即在上下文窗口中消耗约 55K tokens 的工具描述，50 个以上工具在开始工作前即消耗约 77K tokens，极端场景达 134K tokens（Anthropic, 2025）。MIT 2025 AI 智能体指数对 30 个头部智能体系统的评估证实，当前系统普遍缺乏层级化技能组织（MIT, 2025）。这代表了一个根本性的可扩展性挑战：随着可用技能数量增长，向 LLM 展示技能的开销线性增长甚至更差，而选择准确率下降。

现有的工具管理方法可分为三类，各有显著局限。**基于平铺提示的方法**（如早期 LangChain、OpenAI Assistants）将所有可用工具直接列在系统提示中，消耗过多上下文且缺乏结构化记忆。**模块化技能目录**（如 Anthropic Skills、OpenAI GPTs）通过分类降低提示词压力，但缺乏个性化过滤、偏好路由和冷启动自适应。**通用智能体框架**（如 LangGraph、AutoGen）擅长多步任务编排，但未提供针对千级技能规模的专用索引与进化机制。关键在于，这些方法均未同时解决五大核心挑战：可扩展性（O(log N) 检索）、个性化（公共+私人技能层）、可进化性（技能自动晋升与淘汰）、可解释性（可审计的决策轨迹）和 Token 效率（上下文压缩）。

本文提出**个人 AI 技能森林**，首次将 B+ 树多级索引引入智能体技能管理。核心洞察是 B+ 树的层级结构天然映射到"领域-子类别-技能"分类体系，通过逐层缩小搜索空间实现高效检索。系统将技能组织为多棵并行 B+ 树（每个领域一棵），由森林级路由层以 O(log N) 时间将查询导向合适的树。在每个内部节点，代表向量（子节点 embedding 的聚类中心）支持基于相似度的分支选择，而叶子节点存储 10–30 个紧密相关的技能及其依赖和参数元数据。

本文的贡献如下：

1. **架构设计**：提出技能森林架构，包含 12 个互锁机制（M1–M12），协同解决可扩展性、个性化、可进化性、可解释性和 Token 效率问题。

2. **机制设计**：引入五个新颖的操作机制：依赖回溯（M4）自动构建完整执行链；多候选选择（M5）通过用户交互解决意图歧义；参数合并（M6）解决跨层参数冲突；私人技能遮蔽（M7）实现基于路径深度的优先级；角色降维（M9）将 LLM 从多角色推理者转变为纯执行者。

3. **自进化系统**：设计双层反思系统，包含行动反思（四个模式提取器从轨迹中蒸馏可复用操作技能）和思维反思（假设日志→策略抽象→策略挂载的认知循环）。

4. **实验验证**：在 5,000-API 数据集上开展六组综合实验，证明 79.3% Token 节省、21.6pp 自进化成功率提升和跨规模一致的性能表现。

本文其余部分组织如下：第 2 节回顾相关工作；第 3 节介绍技能森林架构；第 4 节详述核心机制；第 5 节描述自进化系统；第 6 节呈现实验设计与结果；第 7 节讨论启示、局限与未来方向；第 8 节总结。

## 2. 相关工作

### 2.1 工具增强语言模型

外部工具与语言模型的集成已快速发展。Toolformer（Schick et al., 2023）证明 LLM 可以通过自监督学习自主使用工具。Gorilla（Patil et al., 2023）专注于提高 API 调用准确率。ToolBench（Qin et al., 2023）建立了包含 16,000+ 真实 API 的基准。API-Bank（Li et al., 2023）提供了工具使用能力的综合评估框架。这些工作推进了工具调用准确率，但主要解决单个工具的正确调用问题，而非高效管理和组织数千个工具。

### 2.2 智能体框架与编排

现代智能体框架为多步任务执行提供基础设施。LangChain（Chase, 2022）引入了思维链工具组合。AutoGen（Wu et al., 2023）支持多智能体对话模式。LangGraph 以有状态图编排扩展了 LangChain。CrewAI 等框架提供基于角色的多智能体协作。然而，这些框架将工具管理视为次要问题，通常依赖平铺工具列表或简单分类，均未提供针对大规模技能检索优化的层级索引。

### 2.3 检索增强生成与工具选择

检索增强生成（RAG）（Lewis et al., 2020）检索相关文档以增强 LLM 生成。近期工作将此范式扩展到工具选择：Toolformer 基于上下文相关性检索工具，GraphRAG（Edge et al., 2024）使用知识图谱进行结构化检索。然而，标准 RAG 方法使用平铺向量索引（如 FAISS、HNSW），缺乏高效领域级路由所需的层级结构。我们的工作从根本上不同，通过引入 B+ 树多级遍历索引，同时提供结构化组织和 O(log N) 检索保证。

### 2.4 层级索引结构

B+ 树是数据库系统的基础数据结构（Comer, 1979），提供 O(log N) 的搜索、插入和删除操作，具有优秀的磁盘 I/O 特性。其叶子节点链表结构天然支持范围查询。虽然 B+ 树在数据库索引中无处不在，但据我们所知，将其应用于 AI 技能管理是新颖的。信息检索中的现有层级方法包括层级聚类（Johnson, 1967）、树结构向量量化和可导航小世界图（Malkov & Yashunin, 2018）。我们的工作独特地将 B+ 树索引与每个树层级的基于 embedding 的相似度搜索相结合，创建了一种利用结构化组织和语义理解的混合结构。

### 2.5 自进化 AI 系统

通过经验改进的 AI 系统概念在机器学习中有深厚根基（Thrun & Schwartz, 1995）。近期关于自我改进智能体的工作包括 Reflexion（Shinn et al., 2023），通过语言反思改进任务表现；Voyager（Wang et al., 2023），通过代码生成在 Minecraft 中积累技能；Self-refine（Madaan et al., 2023），通过自我反馈实现迭代改进。我们的工作通过提出结构化双层反思系统而有所不同：行动反思提取可复用的操作模式（什么有效），思维反思提取元认知策略（如何思考），两层共同贡献于持久化、有组织的技能森林。

### 2.6 AI 助手中的个性化

AI 系统中的个性化范围从简单的偏好存储到复杂的用户建模（Kobsa, 2001）。关于个性化 LLM 的近期工作包括 LaMP（Salemi et al., 2023），以及各种用户特定微调方法。我们的工作引入了一种新颖的个性化机制：基于路径深度的优先级遮蔽（M7），其中挂载在用户树更深处的私人技能自动覆盖相同语义层级的公共技能，无需逐用户模型微调即可提供个性化。

## 3. 系统架构

### 3.1 总体架构

个人 AI 技能森林将技能组织为多棵并行 B+ 树，每棵代表一个功能领域。系统包含三个架构层（图 1）：

1. **森林层（M1）**：五棵独立的 B+ 树，对应五大领域：文档创作、数据分析、通信协作、代码工程、设计创意。
2. **路由层（M2）**：基于根向量的路由机制，将查询导向合适的领域树。
3. **进化层（M10–M12）**：包含双记忆架构和双层反思的自进化系统。

每个 B+ 树节点携带一个代表向量，计算为其所有子节点 embedding 的聚类中心，支持从根到叶子的基于相似度的遍历。树的阶数（每个节点的最大子节点数）设为 32，对于每个领域 1,000 个技能的规模，深度约为 3–4 层。

### 3.2 B+ 树节点结构

B+ 树中的每个节点包含以下字段：

- **keys**：排序的分类路径编码（如 `code/frontend/anim/fly_in`）
- **children**：子节点指针（仅内部节点）
- **apis**：技能列表（仅叶子节点），每个携带依赖规则、互斥规则和优先级元数据
- **representative_vector**：所有后代技能的聚类中心 embedding
- **description**：节点描述，用于向量匹配

代表向量自底向上计算：叶子节点取其技能 embedding 的均值，内部节点取其子节点代表向量的均值。此计算在索引构建时执行一次，技能插入或删除时增量更新。

### 3.3 森林级路由（M2）

路由层维护五个根向量的小型索引（每个领域树一个）。给定查询，系统执行以下步骤：

1. 使用句子转换器模型（all-MiniLM-L6-v2，384 维）编码查询
2. 计算查询 embedding 与各根向量的余弦相似度
3. 选择相似度最高的领域
4. 若 Top-1 与 Top-2 得分差值低于阈值 δ，触发多候选选择机制（M5）

路由步骤的时间复杂度为 O(D)，其中 D 是领域数（通常为 5），相对于总技能数 N 实际为 O(1)。

### 3.4 树内遍历

在选定的领域树内，查询从根到叶遍历：

1. 在每个内部节点，计算查询向量与每个子节点代表向量的余弦相似度
2. 选择相似度最高的子节点
3. 继续直到到达叶子节点
4. 在叶子节点，对存储的 10–30 个技能执行基于 embedding 的精排
5. 返回 Top-K 结果

遍历深度为 O(log_B(N/D))，其中 B 是树的阶数，D 是领域数。对于 N=5,000、D=5、B=32，深度约为 2–3 层。

### 3.5 哈希表分离（M8）

系统维护双轨设计：

- **全局 Skill ID 哈希表**：提供从技能 ID 到元数据的 O(1) 查找
- **B+ 树逻辑结构**：处理所有关系性操作（意图路由、依赖回溯、参数继承）

此分离确保高频查找（如检查技能是否存在）完全绕过树结构，而复杂的关系推理（如"该技能的前置条件是什么？"）利用树的层级组织。

## 4. 核心机制

### 4.1 依赖回溯（M4）

当选中一个技能时，M4 通过递归遍历 `requires` 元数据字段自动追踪其完整依赖链。算法如下：

```
function TraceDependencyChain(skill, visited):
    if skill.id in visited: return []
    visited.add(skill.id)
    chain = []
    for req_id in skill.requires:
        chain.extend(TraceDependencyChain(all_apis[req_id], visited))
    chain.append(skill)
    return chain
```

这确保当用户请求"在 PPT 封面标题上加飞入动画"时，系统返回完整链路：`ppt_planning → ppt_outline → ppt_structure → ppt_cover_page → title_fly_in_animation`，而非仅返回叶子技能。

没有 M4 时，LLM 必须仅从技能描述推断依赖关系。我们的实验表明，LLM 通过语义相似度可推断约 41% 的依赖，但会遗漏传递性和非显而易见的依赖，导致链完整率仅 0.363。

### 4.2 多候选选择（M5）

当路由层检测到歧义（Top-1/Top-2 余弦相似度差值 < δ）时，M5 向用户展示最多 4 个候选领域供选择：

1. 计算查询与所有领域根向量的相似度得分
2. 按得分降序排列领域
3. 若 score₁ - score₂ < δ，将前 4 个领域作为选项 A、B、C、D 展示
4. 用户选择目标领域
5. 在选定领域内进行树内检索

阈值 δ 设为 0.25，基于我们的敏感性分析（实验三），该值在保持合理 Token 开销的同时最大化任务完成率。

### 4.3 参数合并（M6）

技能在多个树层级携带参数：根节点（系统默认）、中间节点（模板参数）、叶子节点（技能特定参数）和用户层（显式覆盖）。M6 使用优先级级联合并这些参数：

**优先级顺序**：用户显式 > 叶子节点 > 中间节点 > 根节点

例如，当用户说"用深色主题做 PPT"时：
- 根节点：`background: white`（系统默认）
- 中间节点：`background: dark`（技术模板）
- 叶子节点：`background: darkblue`（演示风格）
- 用户：`background: dark`（显式覆盖）

M6 按优先级解决所有冲突，产生单一合并参数集。没有 M6 时，冲突参数呈现给 LLM，其正确解决率约为 47%。

### 4.4 私人技能遮蔽（M7）

M7 实现基于路径深度的个性化优先级系统：

- 私人技能存储在用户特定路径下（如 `user/user_001/ppt/theme/dark`）
- 公共技能存储在共享路径下（如 `public/ppt/theme/generic`）
- 当查询同时匹配私人和公共技能时，M7 对私人技能施加优先级加成：若私人技能的相似度得分在最佳公共技能的 0.15 范围内，则选择私人技能

此机制确保用户配置的偏好（如"PPT 始终用深色主题"）一致地覆盖通用默认值，私人技能命中率达 0.667，而没有 M7 时仅为 0.200。

### 4.5 角色降维（M9）

M9 从根本上改变了 LLM 在技能调用流水线中的角色。没有 M9 时，LLM 必须同时充当：

1. **意图理解者**：解析用户请求
2. **技能选择者**：从候选中选择合适的技能
3. **冲突解决者**：解决跨层参数冲突
4. **执行规划者**：确定执行顺序和依赖

有了 M9，结构化流水线（M2 路由 + M4 依赖回溯 + M6 参数合并）预处理所有推理，将 LLM 简化为接收完全解析执行计划的纯**执行者**。此角色降维带来三重收益：

- **Token 效率**：LLM 确认仅需约 30 tokens，而完整推理需 500–2,000 tokens
- **确定性**：结构化操作产生一致结果
- **可组合性**：流水线各阶段可独立优化

## 5. 自进化系统

### 5.1 双记忆架构

系统维护两个记忆层：

- **短期记忆**：当前及近期对话上下文（工作记忆）
- **长期技能记忆**：抽象化、固化后的可执行模式，存储为私人技能（跨会话持久化）

### 5.2 行动反思（第一层）

行动反思通过从交互轨迹中提取可复用操作模式来总结"如何工作"。四个模式提取器对对话日志进行操作：

**提取器 1（重复解决路径）**：识别在多次交互中出现的固定解决路径。例如，若"ImportError → pip install + 修改 import"成功三次，则被提取为可复用模式。

**提取器 2（稳定默认值）**：识别在相似场景中一致选择的默认配置。例如，从重复的 API 超时处理中提取"重试次数默认为 3"。

**提取器 3（固定工作流）**：识别一致成功的有序步骤序列。例如，从数据库连接调试中提取"检查连接字符串 → 检查服务状态 → 检查防火墙 → 开放端口"。

**提取器 4（显式指令捕获）**：监听用户的显式指令，如"记住"或"始终使用"，立即根据指定参数生成私人技能。

### 5.3 思维反思（第二层）

思维反思通过从推理模式中提取元认知策略来总结"如何思考"。三个模块组成认知循环：

**思维日志记录器**：从每次交互中记录 `{假设, 结果, 转折点}` 三元组。

**策略抽象器**：积累三元组并提取"如果 [条件]，则 [行动]"的条件策略。例如，从多次调试交互中提取："如果依赖安装后出错，首先检查版本冲突。"

**策略挂载器**：将提取的策略挂载到技能树的适当分支——更通用的策略挂载在更高层（靠近根），领域特定策略挂载在更低层（靠近叶子）。

### 5.4 运行时模型

两个反思层均作为事件驱动的后台批处理运行，不阻塞主要请求处理。反思触发条件包括：

- 会话完成
- 用户显式反馈（"效果很好"/"换个方式试试"）
- 累积阈值（如 5 次类似交互）

## 6. 实验

### 6.1 实验设置

#### 6.1.1 数据集

我们构建了一个合成 ToolBench 风格数据集，包含 5,000 个 API，均匀分布在 5 个领域（每个领域 1,000 个）。每个 API 包含名称、描述、参数列表、类别标签和依赖规范。数据集使用模板化 API 描述和 all-MiniLM-L6-v2 模型生成的真实 embedding（384 维）。

**依赖图设计**：每个领域采用"双互环+高级层"结构，确保所有 API 都有依赖且最大深度为 3。Level 0（基础层，4 个子类别）形成 2 个互相对依赖对（A↔B, C↔D），深度为 2。Level 1（高级层，2 个子类别）依赖基础层技能，深度为 3。

**测试集**：200 个标注查询（140 个清晰意图 + 60 个跨领域歧义），每个标注了正确领域、正确技能 ID 和歧义标志。

#### 6.1.2 评估指标

- **Accuracy@K**：正确结果出现在 Top-K 中的查询比例
- **MRR（平均倒数排名）**：首个正确结果的平均倒数排名
- **链完整率**：执行链中包含所需依赖的比例
- **冲突消解率**：正确解决的参数冲突比例
- **私人技能命中率**：私人技能被选中而非公共替代品的查询比例
- **Token 消耗**：端到端流水线消耗的总 tokens

#### 6.1.3 统计规范

所有实验遵循严格的统计规范：
- 每个配置 5 次独立运行，报告均值 ± 标准差
- 成对比较使用配对 t 检验（正态分布）或 Wilcoxon 符号秩检验（非正态）
- 主要指标报告 95% 置信区间
- Cohen's d 量化效应量

#### 6.1.4 基线方法

- **Flat ANN（FAISS）**：IVF+PQ 近似最近邻搜索，nlist=50，nprobe=10
- **单棵 B+ 树**：包含所有 5,000 个 API 的混合树，无领域路由
- **技能森林（本方案）**：五棵领域专属 B+ 树，带森林级路由

## 7. 讨论

### 7.1 架构优越性

实验结果从三个维度确立了技能森林的优越性：

**检索质量**：森林通过领域级隔离实现比平铺 ANN 高 28.8% 的 MRR，将每个查询的有效搜索空间从 N 缩减到 N/D 个技能。此优势是结构性的：随着 N 增长，平铺 ANN 必须在所有 N 个技能中区分，而每棵树仅在共享领域上下文的 N/D 个技能中区分。

**Token 效率**：79.3% 的 Token 节省（612→127）主要归因于 M9 角色降维，消除了 LLM 推理依赖、参数和歧义的需要。森林的固定开销（路由+检索+依赖+参数+确认≈127 tokens）不随 N 扩展，而平铺方法的 LLM 推理开销随规模增长。

**链完整率**：从 0.363 到 1.000 的提升证明结构化依赖追踪从根本上比 LLM 推理更可靠。虽然 LLM 可以通过语义相似度推断直接依赖（约 41% 准确率），但传递性依赖需要 M4 提供的显式图遍历。

### 7.2 机制交互效应

消融研究揭示了机制间的重要交互效应：

- **M2 使能 M4**：没有领域路由时，依赖追踪在混合领域树上运行，跨域依赖产生噪声
- **M4 使能 M6**：参数合并需要完整依赖链的知识以正确继承参数
- **M5 补充 M2**：当路由置信度低时，M5 以额外用户交互为代价防止灾难性误路由
- **M9 放大所有机制**：通过消除 LLM 推理开销，M9 使整个结构化流水线在规模上可行

### 7.3 自进化有效性

双层反思系统展示了清晰的学习曲线：

- **行动反思**实现宏平均 F1 0.881，其中 E3（工作流）表现最佳（0.926），因动作关键词明确；E4（显式指令）表现最差（0.795），因需同时匹配触发词+参数词+偏好声明。
- **思维反思**呈现陡峭的改进曲线：Round 1→Round 2 产生 +20.5pp 成功率提升，而 Round 2→Round 3 仅 +1.1pp，表明最具影响力的策略在首轮被捕获。

### 7.4 局限性

**数据局限**：所有实验使用合成数据。虽然模板化 API 描述配合真实 embedding 提供了合理近似，但合成数据的领域区分度可能高于真实世界 API，可能高估路由准确率。

**评估局限**：Token 消耗基于内容长度估算，而非从实际 LLM API 调用测量。链完整率基线使用 embedding 相似度启发式，而非实际 LLM 推理，可能低估或高估真实 LLM 能力。
